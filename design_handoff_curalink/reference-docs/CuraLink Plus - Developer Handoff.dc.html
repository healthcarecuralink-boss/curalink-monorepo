CuraLink Plus — Developer Handoff

Staff/provider app · shares one backend with the CuraLink consumer app · 6 roles · React Native target

Shared backend, one source of truth: CuraLink (consumer) and CuraLink Plus (this app) read/write the same data store. A job accepted here is the same booking the family sees in CuraLink; a nurse's live GPS published from the Visit screens is the same feed the family's tracking map consumes; a prescription/lab report uploaded here lands in the family's Health Records tab. Treat every entity below as shared, not duplicated per app. See the companion CuraLink — Developer Handoff document for the consumer side of this same contract.

3 · Real map integration

All 8 map-bearing screens (Job detail, En route, Live dispatch, Partner-pharmacy locator, Ambulance SOS-request, Pharmacy pickup map, Ambulance transport, Ambulance live map) use Leaflet + OpenStreetMap tiles — genuine pan/zoom/pin rendering, no API key required. Hyderabad-area coordinates are used throughout for realism (Kondapur, Jubilee Hills, Banjara Hills, HITEC City). Swapping to Google Maps SDK in production is a drop-in replacement behind the same marker/route data shape — see leaflet-map.jsx.

4 · Multi-role accounts

A single verified identity can now hold more than one role (e.g. a nurse who is also a Partner Admin) without re-onboarding. Profile → Your roles lists active roles with a + Add role flow that activates instantly (same bank details, same verified documents). A role pill next to the date on Home opens an in-place switcher that swaps the entire active dashboard to any held role — no logout/login round-trip.

5 · Team chat & in-app payouts

Team chat — group/team channels distinct from 1:1 ops support: per-patient Care Team (nurse + doctor + admin on today's active visit), shift-handoff broadcasts, and admin escalation threads. Reached from a header icon on Home and from Profile. The Visit Completed screen now includes an optional handoff-note field that posts directly into the Care Team channel — closing the loop between finishing a visit and briefing whoever covers next.

In-app payouts — linked payout methods (bank account / UPI) with a default + backup badge, method selection at Withdraw, and an itemized Payout history ledger (date, method, amount, status).

Note the deliberate symmetry: a PharmacyOrder created from the nurse-side "Partner pharmacy locator → Send prescription here" flow is the exact record a Pharmacy Partner sees in their Orders queue. An AmbulanceRequest raised from SOS is the exact record an Ambulance Partner sees in their Requests queue. Build these as one table each, not per-app copies.

9 · Key flows to implement first

  1. Job accept → visit → payout (nurse/vet) — the richest loop; vitals/notes/meds/labs/handoff-note all write onto the same Visit record the consumer app's report view reads.
  2. Live GPS publish/consume — Plus's En-route and Ambulance-Transport screens must publish a location stream; CuraLink's tracking screen must be a live subscriber to the same stream, keyed by jobId/requestId. Prototype uses Leaflet/OSM; production can swap to Google Maps SDK behind the same contract.
  3. Pharmacy fulfillment loop — order created (nurse or consumer) → pharmacy accepts with per-item stock/substitute → prepares → rider pickup code → completed. Pickup code should be a real generated OTP server-side, not a static demo value.
  4. Ambulance dispatch loop — request created → nearest on-duty ambulance partners notified → accept → 3-phase transport with live map → hospital handoff → fare settlement.
  5. Multi-role account switching — one identity, N active roles; switching roles must swap dashboard/permissions instantly without a fresh auth round-trip.
  6. Team chat channels — care-team channels should auto-create per active Visit and auto-archive on completion; handoff notes are a message type, not a separate table.
  7. Cura Assistant & ops chat — both call window.claude.complete in the prototype; in production, proxy through your backend so you can inject real patient/job context server-side and keep API keys off-device.

10 · Known-fixed issues (for QA parity)

Bottom tab bar was rendering nowhere in the prototype for its entire build history — the visibility condition referenced a screen key (nav) that was never a real route, so it silently evaluated false on every screen. Fixed by computing showBottomNav from an explicit list of tab-root screens. If you're porting this prototype's logic 1:1, double check any "is this a root/tab screen" condition against your actual route table rather than a magic string.
Template-hole discipline: several early pharmacy/ambulance screens shipped with inline ternaries inside style holes (a conditional expression written directly inside a template placeholder), which this templating engine can't evaluate — properties silently dropped. All such logic was moved into precomputed view-model fields. If your team adopts a similar templating layer, lint for this pattern.
