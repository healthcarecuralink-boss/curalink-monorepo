CuraLink — Developer Handoff

Consumer home-healthcare app · Hyderabad · iOS-first · "Care, comfort, connected."

Shared backend, one source of truth: CuraLink (this app, patient/family-facing) and CuraLink Plus (staff/provider app) read and write the same data store. A visit booked here is the exact job a nurse accepts in Plus; the live map here consumes the exact GPS stream Plus publishes; a prescription written in Plus surfaces here in Health Records. See the companion CuraLink Plus — Developer Handoff document for the staff side of this same contract — §8 there and §7 here must be built as one schema, not two.

1 · Product overview

CuraLink lets Indian families book qualified nurses, doctors, physiotherapists, vets, lab tests, and ambulances to come to their home. The core user is an adult child (25–45) managing care for aging parents, a young parent managing kids' illnesses, or a pet owner — usually opening the app in a stressful, worried moment. The product must feel like a calm, competent friend, not medical chrome.

Primary USP: rapid home care — a verified nurse at your door within the hour, one-tap instant booking, live tracking. This is the reason the app exists and is the hero of the whole experience (see the Home screen's rapid-care hero card, §6).

The prototype covers the entire app: onboarding/auth, home, full booking flow, live-visit tracking, profile/family, history/prescriptions, wallet/payments, support/settings, plus 40+ extended surfaces (vitals, care plan, insurance/claims, lab reports, pharmacy orders, subscriptions, reviews, ambulance booking, video consult, chronic care programs, loyalty rewards, and more — full inventory in §4).

2.2 Typography

Display/headings: Bricolage Grotesque (400–800) — greetings, screen titles, hero numbers, button labels, tight letter-spacing at large sizes. Body/UI/numbers: Inter (400–700). Scale: display 30–42px/800, H1 24–27px/800, H2 19–22px/800, H3 15–16.5px/700–800, body 13.5–16px, caption 10.5–11.5px. Minimum readable size ~10.5px (captions only); hit targets ≥44px.

2.3 Spacing, radii, shadows, icons

8pt spacing scale; screens 20px horizontal padding, cards 14–18px. Radii: cards 16–20px, buttons 12–14px, chips 11–16px, pills 99px, sheets 28px (top corners). Shadows are soft and shallow (card 0 2px 10px rgba(11,29,46,.04); green CTA 0 8px 20px rgba(0,194,124,.22–.28)). Icons: Feather/Lucide line set, stroke-width 1.8–2.4, round caps/joins — no filled/Material icons.

Buttons (primary/secondary/ghost/destructive/icon-only, each with pressed/disabled states), text fields (52px height, phone-prefix and password-reveal variants), cards, status pills (pending/confirmed/en-route/in-progress/completed/cancelled), 5-tab bottom nav with a raised center FAB ("Book"), bottom-sheet modals, empty states, shimmer loading skeletons, dark toast pills. 3 · Global interaction: Cura AI assistant

A persistent floating button (navy pill, pulsing green ring) opens a personalized concierge chat reachable from nearly every screen — it can book visits, track orders, refill meds, and pull records. In production this maps to an LLM tool-calling agent scoped to the authenticated patient's own endpoints (create booking, refill Rx, fetch records, etc.). Hidden on splash/onboarding/auth, the assistant itself, and immersive flows (chat, live tracking, visit-in-progress, ambulance); vertical offset is route-aware to clear bottom nav or sticky CTAs.

5 · Interactions & behavior

  • Navigation: goTo(name, patch?) pushes onto a back-stack; root tabs clear it. 5-tab bottom nav (Home, Bookings, Book-center-FAB, Wallet, Profile).
  • Instant booking (USP): expressBook() preselects nearest nurse, today, earliest window — skips steps 1–4 straight to payment. Wire to a real "find nearest available provider now" backend call.
  • Payment: first attempt intentionally fails (demonstrates the error sheet); retry succeeds. Drive from Razorpay order + webhook in production.
  • Live visit ETA: counts down then flips to Arrived + toast — in production this is a live subscription to the GPS stream CuraLink Plus publishes (see §7).
  • Ratings/reviews, toggles, cart selections are local state in the prototype; persist server-side.

6 · State management (production shape)

Split the prototype's single in-memory state object into: Auth/session (phone/email, OTP, JWT, current user), server data via React Query/RTK Query/Supabase (profile, family, addresses, services, providers, bookings with live status, prescriptions, lab data, vitals, insurance, wallet, pharmacy orders, subscriptions, reviews, notifications, chat), ephemeral UI state (booking-flow selections, cart, toggles, toasts), and realtime subscriptions (visit status + provider GPS, chat, notifications).

7 · Shared backend — core entities

Both apps talk to one backend, one database. Key tables (Postgres/Prisma-style): users (role: patient/provider/admin), family_members, addresses, providers (incl. live location), services/sub_services, bookings (status pending→confirmed→en_route→in_progress→completed|cancelled), visit_events, provider_locations (GPS stream), prescriptions, lab_orders/lab_reports, vitals, insurance_policies/claims, wallet_accounts/transactions, payment_methods (Razorpay), subscriptions, reviews, pharmacy_orders (linked to prescription + fulfilling pharmacy — the exact record CuraLink Plus's Pharmacy Partner role fulfills), ambulance_requests (the exact record an Ambulance Partner accepts), reminders, chat_threads/messages, notifications, plus extended-surface tables for diet plans, referrals, ICU setups, chronic programs, checkup packages, nursing subscriptions, donations, video consults, insurance-card uploads, loyalty accounts, and articles.

8 · Assets

Fonts: Bricolage Grotesque + Inter (Google Fonts). Icons: Feather/Lucide line set — map to lucide-react-native. Map: prototype uses a keyless map embed of Kondapur, Hyderabad; production needs a real Maps SDK + live provider GPS (CuraLink Plus's map screens already run on Leaflet/OSM — same contract, easy to mirror). Content: all names/addresses/prices are realistic Indian seed data (Priya, Rajesh, Meera, Aarav, Simba; Jubilee Hills/Kondapur/Madhapur; ₹ Indian grouping) — safe to reuse as seed data.

9 · Implementation order (suggested)

  1. Backend schema + auth (OTP) + seed data — coordinate field-for-field with the CuraLink Plus team so both apps ship against one contract.
  2. Design tokens + component library (buttons, fields, cards, pills, nav, sheet, toast, skeleton).
  3. Onboarding/auth → Home shell + bottom nav.
  4. Booking flow + instant/express booking + Razorpay.
  5. Live visit tracking + realtime + provider-GPS contract — build this in lockstep with CuraLink Plus's En-route/Transport screens; it is the same stream on both ends.
  6. Profile/family, history, prescriptions, wallet.
  7. Extended surfaces: vitals, care plan, insurance, labs, pharmacy, plans, reviews, ambulance, medical team, second opinion, and the remaining 20+ long-tail screens.
  8. Support/settings/notifications, dark mode.
Build note: the prototype's CuraLink.dc.html is a design reference (inline-styled Design Component), not production code — recreate pixel-accurately in your target codebase (React Native/Expo + TypeScript is the recommended target) rather than porting the file directly.
